## 3. HTML 原型对照方式

### 3.1 启动参考原型

打开：

```text
design-references/warning-default/prototype/index.html#/credit/warning
```

如果浏览器自动化限制 `file://`，使用环境已有静态服务器，不新增项目依赖。例如 Python 已存在时：

```bash
cd design-references/warning-default/prototype
python3 -m http.server 4174
```

### 3.2 启动 React 实现

启动项目开发服务器，记录实际 URL。

### 3.3 同视口对照

对每个页面：

1. 打开参考原型对应 Hash
2. 打开生产实现对应 route / Hash
3. 设置相同视口
4. 截取两张截图
5. 对比：
   - 内容
   - 数据
   - 区块顺序
   - 信息层级
   - 可用操作
   - 弹层行为
   - 状态恢复
   - 视觉意图
6. 不机械复制原型中的偶发字体或截图误差
7. 发现问题立即修复
8. 重新截图

### 3.4 禁止实现方式

验收必须确认 React 实现没有：

- iframe 嵌入原型
- 将原型复制到 `public/` 作为生产页面
- `dangerouslySetInnerHTML` 注入整页
- 在 `useEffect` 中运行原型单文件脚本
- 单一巨型组件复刻全部原型
- 为旧代码兼容牺牲新页面结构

## 19. 控制台、路由和构建

必须：

1. 启动开发服务器并记录URL
2. 访问所有 canonical routes
3. 刷新每个关键子路由
4. 检查：
   - console errors
   - console warnings
   - React warnings
   - failed requests
   - unhandled promise rejections
5. 修复所有本任务引入的问题
6. 运行：

```bash
npm run build
git diff --check
```

HashRouter必须：

- 直接访问Hash路由可渲染
- 刷新不404
- Query保留
- 返回正确

## 20. 修复循环

不得只输出问题报告。

循环执行：

1. 参考原型截图
2. 实现截图
3. 发现差异
4. 判断是业务、交互、视觉还是技术问题
5. 直接修复
6. 重启或热更新
7. 重新点击
8. 重新截图
9. 再次验证

停止条件：

- 所有阻断项通过
- 所有主要视觉和交互问题通过
- 所有数据一致
- 所有关键视口通过
- 构建通过
- 控制台干净

“基本完成”“大致一致”“后续优化”不是停止条件。

---

## 21. 最终输出

最终输出三个清单。

### A. 已通过

按以下分类：

- 页面
- 流程
- 视口
- 业务数据
- 状态语义
- 交互
- AI
- 无障碍
- 路由
- 构建

### B. 已发现并修复

每项包括：

- 问题
- 影响页面
- 修改文件
- 修复方式
- 重新验证结果
- 对应截图路径

### C. 仍有差异

每项包括：

- 差异
- 与原型或规范的关系
- 原因
- 是否影响业务和验收
- 是否需要用户确认

不得隐藏未完成项，也不得将未验证项写成已通过。
